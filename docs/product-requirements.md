## 三、需求背景与目标

### 3.1 产品现状

基金用户可能在支付宝、天天基金、银行等不同平台购买基金。不同平台的持仓相互隔离，用户难以在一个视图中了解整体持仓、累计收益和盘中变化。

部分产品支持手动录入、截图识别、特定文件导入或净值估算，但用户迁移大量持仓时仍可能遇到重复录入、格式不统一、数据来源不清楚和结果难以复核的问题。

### 3.2 用户问题

- 多个平台的基金持仓无法统一管理和筛选。
- 首次迁移多只基金时，逐只录入成本较高。
- 普通用户知道支付金额和购买日期，却不理解确认份额和成本计算。
- 盘中估值、官方净值和平台确认数据容易被混为一谈。
- 历史记录发生错误时，直接修改会导致后续成本和收益无法追溯。
- 用户缺少开放、可备份、可迁移的数据协议。

### 3.3 产品目标

Fund Keeper 面向在一个或多个平台持有基金的普通投资者，通过金额优先的单笔录入、标准 JSON 批量迁移和跨账户汇总，降低持仓建立与维护成本，并展示明确标注的盘中预估收益和基于官方净值计算的收益。

系统必须保留计算所需的份额、成本和交易状态，但前台优先展示金额、收益和持有时间。产品不执行基金交易，不保存第三方平台登录凭证，不提供基金推荐、买卖建议或收益保证。

### 3.4 V1 体验目标

- 用户无需计算基金份额即可建立持仓。
- 用户可以通过单笔录入或 JSON 批量建立多只基金持仓。
- 批量导入必须先预检，用户确认后才正式写入。
- 成功建立持仓后立即进入首页，看到可用的持仓和收益数据。
- 估算数据必须说明来源、状态和时间，不伪装成平台确认结果。
- 首次建立持仓所需时间作为上线后观察指标，V1 不设置未经验证的固定时长目标。

### 3.5 数据指标

| 指标 | 口径 | V1 目标 |
|---|---|---|
| 注册完成率 | 完成邮箱验证用户数 / 开始注册用户数 | 上线后建立基线 |
| 持仓建立率 | 至少建立一笔有效持仓用户数 / 注册用户数 | 上线后建立基线 |
| 手动录入成功率 | 成功创建记录次数 / 有效提交次数 | 上线后建立基线 |
| JSON 预检通过率 | 预检通过批次 / 预检批次 | 上线后建立基线 |
| JSON 提交成功率 | 成功提交批次 / 确认提交批次 | 持续监控异常 |
| 盘中估值覆盖率 | 有有效估值的持仓基金数 / 持仓基金数 | 上线后建立基线 |
| 估值新鲜度 | `LIVE` 数据量 / 盘中估值数据量 | 上线后建立基线 |
| 持仓校准率 | 完成校准的估算持仓 / 估算持仓 | 上线后建立基线 |
| 次日回访率 | 次日打开首页用户数 / 完成持仓用户数 | 上线后建立基线 |

## 四、目标用户与核心场景

### 4.1 首要目标用户

V1 首要面向普通基金投资者。这类用户可能在一个或多个平台购买基金，日常更关心：

- 一共支付了多少钱；
- 当前持有资产大约是多少；
- 当前和累计赚了或亏了多少钱；
- 今天盘中大约赚了或亏了多少钱；
- 每只基金连续持有了多少天。

用户通常不会主动关注基金份额、单位成本和确认净值的计算过程。系统可以在后台保存和使用份额，但前台应优先使用金额化、容易理解的方式展示。

### 4.2 非首要用户

V1 不主要面向：

- 需要复杂资产配置、归因和风险模型的专业投资者；
- 高频进行基金交易的用户；
- 需要直接在产品内完成申购或赎回的用户；
- 需要同时管理股票、ETF、LOF、债券或数字货币的用户；
- 需要基金推荐、跟投、社区和投资建议的用户。

### 4.3 核心使用场景

1. **首次建立持仓**：用户通过单笔录入或标准 JSON 快速建立多只基金持仓。
2. **每天查看收益**：用户统一查看当前资产、累计收益、盘中预估收益和持有天数。
3. **持续维护持仓**：用户记录后续买入、部分卖出、全部卖出和平台确认结果。
4. **迁移与备份**：用户将持仓或完整交易流水导出为标准 JSON。
5. **修正历史错误**：用户在看到影响预览后更正确认交易，系统保留完整审计链。

## 五、需求范围

### 5.1 V1 范围内

#### 用户与账户

- 邮箱验证码注册、邮箱密码登录和邮箱找回密码。
- 注册成功后自动创建“默认账户”。
- 创建、修改、归档多个基金平台账户。
- 用户数据严格隔离。

#### 基金范围

- 人民币计价的境内场外开放式基金。
- 股票型、混合型和指数型基金。
- 通过基金代码查询名称和基础资料。

#### 持仓录入

- 逐只手动录入买入、部分卖出和全部卖出。
- 持仓快照 JSON 导入。
- 交易流水 JSON 导入。
- JSON 文本粘贴和本地 `.json` 文件选择。
- 预检、用户确认、整批提交、幂等和整体回滚。
- JSON 持仓快照和交易流水导出。

#### 持仓与收益

- 全部账户聚合和单账户筛选。
- 当前持仓成本、当前市值、当前持有收益、已实现收益和累计收益。
- 连续持有天数。
- 盘中预估净值、预估收益、估值时间和数据状态。
- 基于最近官方净值的降级展示。

#### 数据维护

- 查看交易历史和导入批次。
- 修改或取消待确认、系统估算记录。
- 标记外部交易未完成。
- 校准估算持仓。
- 对确认记录执行影响预览、冲正和替代。

### 5.2 V1 范围外

- 真实基金申购、赎回、资金托管和自动交易。
- 保存支付宝、银行等第三方平台的登录凭证。
- 自动登录第三方平台同步持仓。
- OCR 截图识别和 CSV 转换；未来可转换为标准 JSON。
- QDII、货币基金、场内 ETF、LOF、股票、债券和数字货币。
- 基金推荐、收益承诺、投资建议、社区、排行榜和跟投。
- 组合诊断、风险测评和专业资产配置。
- React Web 用户端、运营后台、微服务和 Kubernetes。
- “盘中预估收益等于最终官方收益”的任何承诺。

## 六、产品核心契约

### 6.1 身份、对象与隔离

| 对象 | 权威作用域 | 说明 |
|---|---|---|
| 用户数据 | 当前登录用户 | 用户只能访问自己的资产数据 |
| 平台账户 | 用户 + 平台账户 | 同一用户可创建多个账户 |
| 当前持仓 | 平台账户 + 基金 | 同一基金在不同账户分别计算 |
| 交易记录 | 平台账户 + 基金 | 买入、卖出和更正必须指定账户 |
| 导入批次 | 当前用户 | `batchId` 在用户范围内唯一 |
| 聚合持仓 | 当前用户 + 基金 | 只用于展示，不覆盖账户明细 |

服务端必须根据已验证登录态确定用户身份，不得相信客户端提交的 `userId`。同一基金在多个平台分别记账，首页可以聚合，底层交易和成本不能合并。

串数据反例：

- 用户 A 不得查看或修改用户 B 的账户和持仓。
- 伪造其他用户的账户 ID 必须被拒绝。
- 支付宝账户中的卖出不得减少天天基金账户中的同名基金。
- 相同外部 `batchId` 不得跨用户共享处理结果。

### 6.2 数据权威等级

| 等级 | 数据来源 | 示例 | 用户状态 |
|---|---|---|---|
| 1 | 用户录入的平台最终确认结果 | 确认份额、实际到账金额 | 已确认 |
| 2 | 官方基金数据 | 官方净值、净值日期 | 官方数据 |
| 3 | 系统计算结果 | 根据金额、净值和费率推算份额 | 系统估算 |
| 4 | 第三方盘中估值 | 预估净值、预估涨跌幅 | 盘中预估 |

平台最终确认结果是交易确认的最高优先级数据。没有平台确认结果时，系统可以使用官方数据和用户输入进行估算，但必须标记来源。盘中估值只用于预估展示，不得覆盖官方净值、确认份额和历史交易。

系统必须保留数据来源、业务时间和计算时间，不能只保存一个没有来源说明的最终数字。

### 6.3 交易与持仓状态

| 状态 | 用户含义 | 是否参与持仓 |
|---|---|---|
| `PENDING` | 等待净值或确认信息 | 不作为最终确认数据 |
| `ESTIMATED` | 系统已经推算 | 是，必须标记预估 |
| `CONFIRMED` | 用户提供平台确认结果 | 是 |
| `NEEDS_CALIBRATION` | 新快照与系统结果存在差异 | 保留当前结果并提示 |
| `CANCELLED` | 用户确认外部交易未完成 | 否 |
| `REVERSED` | 已被后续更正冲销 | 否 |

核心流转：

```text
PENDING
├── 官方数据可用 → ESTIMATED
├── 用户补充平台结果 → CONFIRMED
└── 用户确认外部交易未完成 → CANCELLED

ESTIMATED
├── 用户补充平台结果 → CONFIRMED
├── 快照发现差异 → NEEDS_CALIBRATION
└── 用户确认外部交易未完成 → CANCELLED

CONFIRMED
└── 用户更正 → REVERSED + 新替代记录
```

系统不连接真实交易平台，不能因为长期缺少确认数据就自动认定买入或卖出失败。

### 6.4 汇总展示原则

- 首页允许 `CONFIRMED`、`ESTIMATED` 和 `NEEDS_CALIBRATION` 持仓共同参与金额汇总。
- `PENDING` 交易单独展示，不计入当前持仓、收益和资产金额。
- 汇总包含 `ESTIMATED` 或 `NEEDS_CALIBRATION` 数据时，显示“包含预估数据”。
- 用户可以查看具体哪些基金和交易尚未确认或需要校准。
- 加载失败或数据缺失不能用 0 替代。
- 用户刷新页面的时间不能冒充估值数据时间。

### 6.5 金额与收益口径

- 买入金额表示用户实际支付的现金总额，包含申购费用。
- 卖出金额在确认前表示预计到账金额，确认后表示扣除赎回费用后的实际到账金额。
- 当前持仓成本表示卖出后仍留在当前持仓中的剩余成本。
- 当前持有收益表示当前市值减去当前持仓成本。
- 已实现收益表示历史卖出净到账金额减去相应移出成本。
- 累计收益表示当前持有收益与已实现收益之和；分红口径待 2.4 决定后补充。
- V1 使用移动平均成本法。

部分卖出时：

```text
移出成本 = 卖出份额 × 卖出前平均单位成本
剩余成本 = 卖出前剩余成本 - 移出成本
已实现收益 = 卖出净到账金额 - 移出成本
```

### 6.6 连续持有天数

- 从当前持有周期中最早一笔有效买入的确认日期开始计算。
- 加仓和部分卖出不重置。
- 全部清仓时结束当前持有周期。
- 清仓后再次买入时开始新的持有周期。
- 跨账户聚合时，使用当前仍持有该基金的账户中最早持有日期。

### 6.7 安全与隐私

- 不保存明文密码和第三方平台登录凭证。
- 登录、验证码和密码重置需要频率控制。
- 密码重置成功后，其他设备的旧登录状态失效。
- 日志不得输出密码、Token、验证码、完整 JSON 和完整资产明细。
- 埋点不得上传邮箱全文、购买金额、收益金额、用户备注和完整导入内容。
- JSON 导出前提示文件包含个人资产信息。

## 七、功能详细说明

### 7.1 注册与登录

| 项目 | 产品契约 |
|---|---|
| 注册方式 | 邮箱验证码 + 密码 |
| 注册结果 | 创建用户并自动创建“默认账户” |
| 重复注册 | 已注册邮箱不能再次创建账户 |
| 登录方式 | 邮箱 + 密码 |
| 登录错误 | 不区分邮箱不存在和密码错误 |
| 找回密码 | 邮箱验证码验证后设置新密码 |
| 重置结果 | 其他设备上的旧登录状态失效 |
| 退出登录 | 清除本机登录状态，不删除服务端资产 |
| 开发环境 | 可以使用模拟邮件发送器 |
| 生产环境 | 使用可替换的第三方邮件服务 |

异常规则：

- 邮件发送失败时不得创建已完成注册的正式用户。
- 验证码只能使用一次，并具有有效期和发送频率限制。
- 注册或登录失败时保留邮箱输入，不保留明文密码。
- 用户身份失效时要求重新认证，不静默切换为其他用户。

### 7.2 基金平台账户

#### 7.2.1 默认账户

- 用户注册成功后自动创建“默认账户”。
- 用户无需先理解平台账户即可直接录入或导入。
- 用户可以重命名默认账户并修改平台类型。
- 首页默认聚合全部有效账户。

#### 7.2.2 创建与修改

| 字段 | 规则 |
|---|---|
| 账户名称 | 用户填写；同一用户的有效账户不能重名 |
| 平台类型 | 支付宝、天天基金、银行、其他 |
| 默认选择 | 用户未选择时使用默认账户 |
| 修改范围 | 名称和平台类型 |

#### 7.2.3 归档与删除

- 存在当前持仓、待确认交易或待处理导入的账户不能删除或归档。
- 已清仓但存在历史交易的账户只能归档。
- 归档账户不允许新增交易，但历史交易和已实现收益仍可查看。
- 完全没有业务数据的空账户可以二次确认后删除。
- 用户必须至少保留一个有效账户。
- V1 不支持账户间直接迁移持仓。

### 7.3 手动交易录入

#### 7.3.1 手动买入

用户输入：

| 字段 | 必填 | 说明 |
|---|---|---|
| 平台账户 | 是 | 默认选中默认账户 |
| 基金代码 | 是 | 系统查询基金名称和支持范围 |
| 买入金额 | 是 | 实际支付总金额 |
| 提交日期 | 是 | 在外部平台提交的日期 |
| 15:00 前/后 | 是 | `BEFORE_15` 或 `AFTER_15` |
| 平台确认份额 | 否 | 高级校准字段 |
| 备注 | 否 | 用户自用 |

系统负责：

1. 判断基金是否属于 V1 支持范围；
2. 根据日期、15:00 前后、周末和节假日计算有效交易日；
3. 查询对应官方净值；
4. 查询或使用可说明来源的费率；
5. 计算预计份额；
6. 创建交易记录并更新相应持仓；
7. 根据数据来源设置状态。

异常规则：

- 基金代码不存在或不支持时阻止提交。
- 金额必须大于 0。
- 提交日期不能晚于当前日期。
- 已归档账户不能新增交易。
- 基金数据不可用时不伪造名称、净值和份额。
- 请求超时且结果未知时先查询原请求，不直接重复创建。
- 创建失败时保留除密码等敏感信息以外的表单现场。

#### 7.3.2 手动卖出

用户选择“部分卖出”或“全部卖出”。

| 字段 | 部分卖出 | 全部卖出 |
|---|---|---|
| 平台账户 | 必填 | 必填 |
| 基金 | 必填 | 必填 |
| 预计到账金额 | 必填 | 不要求 |
| 提交日期 | 必填 | 必填 |
| 15:00 前/后 | 必填 | 必填 |
| 实际到账金额 | 可后补 | 可后补 |
| 平台确认份额 | 可后补 | 不强制 |
| 备注 | 可选 | 可选 |

规则：

- 当前账户必须存在该基金持仓。
- 提交后展示待确认卖出，不立即删除原持仓。
- 部分卖出根据公开数据计算预计减少份额，未校准前保持预估状态。
- 全部卖出确认后持仓归零，并结束本轮连续持有周期。
- 同一用户、同一平台账户、同一基金，同一时间最多存在一笔待确认卖出。
- 部分卖出金额明显超过当前预估市值时阻止提交，并建议使用全部卖出。
- 任何计算不得产生负份额。
- 外部交易被撤销或未受理时，由用户标记“交易未完成”。

### 7.4 JSON 导入与导出

#### 7.4.1 定位与入口

JSON 是统一批量协议和高级入口。V1 支持粘贴 JSON 文本和选择本地 `.json` 文件，并提供协议说明和示例。未来的 OCR、CSV 或 AI 识别结果可以转换为同一协议，但不属于 V1 功能。

#### 7.4.2 持仓快照协议

```json
{
  "schemaVersion": "1.0",
  "importType": "POSITION_SNAPSHOT",
  "batchId": "snapshot-20260724-001",
  "snapshotMode": "FULL_ACCOUNT",
  "account": {
    "name": "我的支付宝",
    "platform": "ALIPAY"
  },
  "snapshotAt": "2026-07-24T14:30:00+08:00",
  "positions": [
    {
      "fundCode": "005827",
      "costAmount": 10000.00,
      "currentAmount": 10326.40,
      "holdingStartDate": "2026-06-01"
    }
  ]
}
```

`snapshotMode` 必须显式声明：

- `FULL_ACCOUNT`：表示目标账户在 `snapshotAt` 时的全部持仓。未列出的现有基金视为清仓，但预检必须醒目提示。
- `PARTIAL`：只新增或校准列出的基金，未列出的持仓保持不变。

快照字段语义：

| 字段 | 必填 | 说明 |
|---|---|---|
| `fundCode` | 是 | V1 支持范围内的基金代码 |
| `costAmount` | 是 | 快照时仍在持仓中的成本，不是累计买入金额 |
| `currentAmount` | 是 | 外部平台在 `snapshotAt` 展示的持有金额 |
| `holdingStartDate` | 是 | 当前连续持有周期的开始日期 |
| `confirmedShares` | 否 | 平台确认份额；存在时优先使用 |

缺少 `confirmedShares` 时，系统根据 `currentAmount` 和快照时可获得的净值推算份额，并将持仓标记为 `ESTIMATED`。净值不可用时预检失败，不得只保存金额后伪造可持续计算的持仓。客户端不得通过 JSON 指定 `CONFIRMED` 等内部状态。

系统不得根据记录数量猜测快照范围。快照通过校准记录更新当前结果，不删除旧交易。

#### 7.4.3 交易流水协议

```json
{
  "schemaVersion": "1.0",
  "importType": "TRANSACTION_BATCH",
  "batchId": "transactions-20260724-001",
  "account": {
    "name": "我的支付宝",
    "platform": "ALIPAY"
  },
  "transactions": [
    {
      "rowId": "row-001",
      "fundCode": "005827",
      "type": "BUY",
      "amount": 5000.00,
      "submittedDate": "2026-06-01",
      "submittedPeriod": "BEFORE_15",
      "confirmedShares": 1234.56
    }
  ]
}
```

流水字段语义：

| 字段 | 适用范围 | 说明 |
|---|---|---|
| `rowId` | 全部 | 批次内唯一，用于定位错误和防止行重复 |
| `fundCode` | 全部 | V1 支持范围内的基金代码 |
| `type` | 全部 | `BUY` 或 `SELL` |
| `amount` | 买入 | 用户实际支付的现金总额，包含申购费 |
| `sellMode` | 卖出 | `PARTIAL` 或 `FULL` |
| `expectedAmount` | 部分卖出 | 预计到账金额 |
| `actualReceivedAmount` | 卖出，可选 | 平台最终确认的净到账金额 |
| `submittedDate` | 全部 | 用户在外部平台提交的日期 |
| `submittedPeriod` | 全部 | `BEFORE_15` 或 `AFTER_15` |
| `confirmedShares` | 买入或部分卖出，可选 | 平台最终确认的份额 |
| `confirmedDate` | 已取得平台结果时可选 | 平台实际确认日期 |

全部卖出不要求用户填写份额；部分卖出必须提供 `expectedAmount`，取得平台结果后可同时提供实际到账金额和确认份额。服务端根据字段完整性、官方数据和计算结果确定记录状态，JSON 不接受调用方直接指定内部状态。

协议不得要求用户知道数据库内部 `accountId`。快照和流水含义不同，系统不得根据字段组合猜测 `importType`。

#### 7.4.4 账户匹配与创建

- 系统按当前用户范围内的账户名称和平台类型匹配目标账户。
- 账户不存在时，预检明确显示“将创建新账户”。
- 用户确认后，账户创建和数据写入在同一业务事务中完成。
- 未经确认不得静默创建账户。

#### 7.4.5 两阶段导入

```text
提交 JSON
→ JSON 语法校验
→ Schema 校验
→ 业务规则校验
→ 展示逐行预检结果
→ 用户确认
→ 整批事务写入
→ 返回最终结果
```

预检至少展示：

- 协议版本、导入类型、批次编号和目标账户；
- 总数、可导入数、警告数和错误数；
- 每条错误对应的行、字段和用户可理解原因；
- 将新增、更新、保持、清仓和校准的持仓；
- 将创建的平台账户；
- 早于现有快照而需要历史重建的交易。

规则：

- 预检阶段不写正式业务数据。
- 存在错误时不能确认提交。
- 用户修改 JSON 后必须重新预检。
- 确认只能提交最后一次成功预检的原始内容。
- 任一写入失败时整批回滚。
- `batchId` 在当前用户范围内唯一。
- 相同 `batchId` 重复提交时返回原结果，不重复记账。

#### 7.4.6 快照时间边界与历史重建

持仓快照会在目标账户上建立时间边界。正常追加交易必须晚于最近一次生效快照。早于快照的历史交易不能直接追加，必须：

1. 从最早交易重新计算；
2. 与当前快照比较；
3. 展示成本、份额和收益差异；
4. 用户确认后替代期初基础或生成校准记录；
5. 任一步失败时保持原持仓不变。

#### 7.4.7 JSON 导出

用户可以导出：

- 当前持仓快照或完整交易流水；
- 全部账户或指定账户；
- 协议版本、导出时间和数据范围。

导出文件不得包含密码、密码摘要、登录令牌、验证码、权限信息和服务密钥。导出协议使用新的可移植批次标识，不复用原始内部导入批次；同一导出文件重复导入时仍遵守幂等规则。

### 7.5 持仓首页与基金详情

#### 7.5.1 首页汇总

首页默认聚合当前用户全部有效账户，也允许按账户筛选。汇总区域至少展示：

- 当前持仓成本；
- 当前市值；
- 当前持有收益；
- 已实现收益；
- 累计收益及收益率；
- 今日盘中预估收益；
- 数据状态和估值时间。

任一数据包含估算结果时显示“包含预估数据”。部分基金未计入汇总时必须明确提示。

#### 7.5.2 基金列表

相同基金在首页聚合为一张卡片，但底层账户记录保持独立。基金卡片至少展示：

- 基金名称和代码；
- 当前持有金额；
- 当前持有收益和收益率；
- 今日盘中预估收益；
- 连续持有天数；
- 数据状态。

点击基金后，用户可以查看各平台账户明细、交易历史、已实现收益、待确认记录和校准入口。

#### 7.5.3 空状态与失败

- 无持仓时展示“添加一只基金”和“批量导入”。
- 只有待确认买入时展示待确认记录，不伪造当前收益。
- 部分基金加载失败时，其他基金继续展示。
- 加载失败时可以保留最近一次成功结果，但必须展示真实数据时间和失败提示。
- 不得用 0 元收益表示加载失败或数据缺失。

### 7.6 盘中预估收益

#### 7.6.1 数据状态

| 状态 | 用户文案 | 行为 |
|---|---|---|
| `LIVE` | 盘中预估 | 正常展示估值和时间 |
| `DELAYED` | 更新延迟 | 继续展示估值，但明确提示延迟 |
| `STALE` | 数据已过期 | 降级使用最近官方净值 |
| `MARKET_CLOSED` | 休市 | 不展示新的今日预估收益 |
| `UNAVAILABLE` | 暂无数据 | 不用 0 替代 |

`priceType` 另行区分 `ESTIMATED`（盘中估值）和 `OFFICIAL`
（正式净值）。每项结果至少包含净值、预估涨跌幅、数据时间、抓取时间、
状态、价格类型和数据来源。

#### 7.6.2 刷新与时效

- 交易时段内，后端最多每 60 秒更新一次活跃基金估值。
- Flutter 前台每 30 秒获取一次后端最新可用结果。
- 客户端轮询不代表每次请求第三方数据源。
- 估值抓取时间超过 90 秒时变为 `DELAYED`。
- 估值时间超过 3 分钟时变为 `STALE`。
- 用户下拉刷新只重新读取后端最新结果，不强制请求第三方。
- 页面展示估值数据时间，不用用户刷新时间替代。
- 官方净值可用后，使用官方数据重新计算。

#### 7.6.3 异常与降级

- 单只基金估值失败不影响其他基金。
- 部分基金失败时，总预估收益标记“部分基金未计入”。
- 第三方失败时优先返回缓存结果，并根据时间设置状态。
- 无缓存时回退到最近官方净值。
- 官方净值也不存在时显示 `UNAVAILABLE`。
- 休市时显示最近官方净值日期和累计收益，不把上个交易日估值冒充今日数据。
- 盘中预估不得覆盖历史官方净值。

### 7.7 取消、校准与更正

#### 7.7.1 标记交易未完成

- `PENDING` 或 `ESTIMATED` 记录可以标记为“交易未完成”。
- 操作前展示该记录将不再参与持仓和收益计算，并要求二次确认。
- 操作后状态为 `CANCELLED`，保留原记录和取消时间。
- `CONFIRMED` 记录不能取消，只能更正。

#### 7.7.2 估算数据校准

- 用户可以补充平台确认份额、实际到账金额或新的持仓快照。
- 仅根据卖出金额推算的份额不得标记为平台确认数据。
- 新快照与系统结果存在差异时进入 `NEEDS_CALIBRATION`。
- 用户确认校准前，系统保留当前持仓并展示差异。

#### 7.7.3 已确认记录更正

已确认交易不得直接覆盖或物理删除。用户选择“更正交易”后，系统必须：

1. 展示原值和新值；
2. 预估对当前持仓、成本、已实现收益、累计收益和后续交易的影响；
3. 要求用户二次确认和填写更正原因；
4. 创建冲正记录和替代记录；
5. 从被更正交易日期开始重新计算；
6. 保留原记录并标记 `REVERSED`。

冲正、替代和重新计算必须整体成功或整体失败。重新计算期间不得向用户展示一半新、一半旧的数据。

## 八、状态与异常矩阵

### 8.1 导入批次状态

| 状态 | 含义 | 允许操作 |
|---|---|---|
| `DRAFT` | 已提交内容，尚未预检 | 发起预检 |
| `PREFLIGHT_FAILED` | 语法、Schema 或业务规则失败 | 修改后重新预检 |
| `READY_TO_COMMIT` | 预检通过 | 用户确认 |
| `COMMITTED` | 整批成功 | 查看结果、重复请求返回原结果 |
| `COMMIT_FAILED` | 写入失败且已整体回滚 | 排查后重新发起 |

### 8.2 关键异常处理

| 场景 | 产品行为 |
|---|---|
| 请求超时，结果未知 | 查询原请求，不盲目重复写入 |
| 基金数据暂不可用 | 保留输入，说明原因，不伪造结果 |
| 第三方估值失败 | 缓存降级、官方净值降级或不可用 |
| JSON 某一行错误 | 指明行和字段，整批不能确认 |
| JSON 提交时一行写入失败 | 整批回滚 |
| 账户无权限或已归档 | 拒绝操作 |
| 更正重新计算失败 | 保留原始持仓和收益 |
| 邮件服务失败 | 暂停需要邮件验证的操作，不影响已有用户查看资产 |
| Redis 不可用 | 正式持仓仍可查询，盘中估值和缓存能力降级 |

| C-02 | 更正确认记录 | 影响预览后生成冲正和替代记录 |
| C-03 | 冲正执行失败 | 原持仓和收益保持不变 |
| E-01 | 导出交易流水 | 文件可以重新通过预检 |
| E-02 | 检查导出文件 | 不包含密码、Token、验证码或密钥 |

## 十一、发布、兼容与回滚

### 11.1 发布顺序

1. 后端先支持身份、账户、状态、JSON 协议和估值状态。
2. 完成基金基础数据、官方净值、交易日历和估值数据源联调。
3. 发布 Flutter 注册、录入、导入、首页、详情和更正功能。
4. 开启公开注册和生产邮件发送。
5. 观察导入失败率、估值覆盖率、降级率和异常日志。

### 11.2 JSON 兼容

- 每份 JSON 必须包含 `schemaVersion`。
- 不支持的主版本必须拒绝并说明原因。
- 新增可选字段不能破坏旧版本导入。
- 协议升级必须提供旧版本迁移方式。
- 系统不得根据字段组合猜测协议版本或导入类型。

### 11.3 回滚原则

- 客户端回滚不得删除服务端持仓。
- 后端回滚不得把 `COMMITTED` 批次重新变成未提交。
- 已确认交易、冲正和校准记录必须保留。
- 估值能力下线时回退到最近官方净值，核心持仓仍可查看。
- 邮件服务异常时暂停新注册和密码重置，不影响已有用户查看资产。
- 回滚不得产生部分导入、负份额或跨用户数据。

## 十二、技术交付边界

本 PRD 锁定用户行为、数据权威、状态、范围、异常和验收，不锁定：

- API 路径和 DTO 类名；
- 数据库表名和字段名；
- JWT、Session 或其他登录实现；
- MySQL 索引、Redis Key 和锁实现；
- 具体基金数据供应商；
- 邮件服务供应商；
- 错误码和埋点事件名；
- Flutter 状态管理框架。

技术方案必须满足：

- 身份从已验证登录态获取；
- 单笔操作和批次操作具备幂等能力；
- 交易、持仓、冲正和导入保持事务一致性；
- Redis 不是交易和持仓的唯一事实来源；
- 第三方服务故障不拖垮正式持仓查询；
- 所有预估和确认状态可追溯。
